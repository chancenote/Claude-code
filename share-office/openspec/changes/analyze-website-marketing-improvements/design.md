## Context

NextCowork는 전주 기반 "Space × AI" 공유오피스 통합 컨설팅 서비스로, 단일 정적 랜딩 페이지(`index.html` + `css/style.css` + `js/main.js`)를 통해 리드를 획득하고 있다. 페이지 구조는 Hero → Pain Points → Services → Why Us → Process → Credibility → FAQ → Contact 순서이며, 외부 라이브러리 없이 바닐라 HTML/CSS/JS로 구성되어 있다. 백엔드 없이 프론트엔드만 존재하므로 폼 제출은 클라이언트 사이드 시뮬레이션(`setTimeout` 후 성공 메시지)으로 처리되고 있다.

현재 한계:
- 마케터 관점의 퍼널 진단 프레임이 없어 개선 우선순위를 객관적으로 잡을 수 없다.
- CTA가 여러 곳에 분산되어 있지만(nav 상단, Hero 2개, 서비스 배너, Contact 섹션) 일관된 전환 흐름으로 설계되지 않았다.
- 폼 input에 `name` 속성이 없고, 실제 전송 엔드포인트가 없어 리드 수집이 작동하지 않는다.
- 신뢰 요소(후기 2건, 실적 수치, 기관 로고 7개)가 존재하지만 전환 경로 상의 전략적 배치가 되어 있지 않다.
- 전화번호가 플레이스홀더(`010-0000-0000`)이고, 카카오톡 링크가 일반 `open.kakao.com`으로 실제 상담 채널과 연결되지 않는다.

이해관계자: 사이트 운영자(NextCowork 대표), 잠재 컨설팅 고객(공유오피스 창업/운영 희망자).

## Goals / Non-Goals

**Goals:**
- AITA(Attention → Interest → Trust → Action) 퍼널 프레임으로 현재 페이지를 구조적으로 진단한다.
- 섹션별 개선 기회를 Impact(전환 기여도) × Effort(구현 난이도) 매트릭스로 우선순위화한다.
- 개선 항목을 섹션 단위(Hero, Services, Credibility 등)와 크로스커팅(CTA 일관성, 폼 UX, 모바일)으로 분리해 병렬 실행 가능하게 구성한다.
- 이벤트 기반 + 스크롤 깊이 측정 전략을 정의해 개선 효과를 정량적으로 검증한다.

**Non-Goals:**
- 백엔드 구축(폼 전송 서버, DB, CRM 연동) — 프론트엔드 개선 방향 정의까지만 범위.
- SEO 기술 감사(메타 태그, 사이트맵, 구조화 데이터) — 별도 change로 다룬다.
- 브랜드 리디자인(로고, 컬러 시스템, 폰트 변경) — 기존 디자인 시스템 내에서 개선한다.
- 다국어/다페이지 확장 — 현재 단일 한국어 랜딩 페이지 범위로 한정한다.

## Decisions

### D1. 분석 프레임: AITA 퍼널 모델 채택

**선택:** Attention → Interest → Trust → Action 4단계 퍼널을 진단 기준으로 사용한다.

**대안 검토:**
- AIDA(Attention-Interest-Desire-Action): B2C 제품 판매에 최적화. 고관여 B2B 컨설팅 서비스에는 Trust 단계가 명시되지 않아 부적합.
- AARRR(해적 지표): SaaS/앱 기반 지표. 단일 랜딩 페이지 분석에는 과도하게 복잡.

**근거:** NextCowork는 B2B 고관여 서비스(컨설팅 수천만 원 단위)이므로 신뢰 구축(Trust)이 전환의 핵심 병목이다. AITA는 Trust를 명시적 단계로 분리해 신뢰 요소(후기, 실적, 비교표, 기관 로고) 진단에 집중할 수 있다.

### D2. 우선순위화 방법: Impact-Effort 매트릭스

**선택:** 각 개선 항목을 Impact(High/Medium/Low) × Effort(Low/Medium/High)로 분류하고, Quick Win(High Impact + Low Effort)부터 실행한다.

**대안 검토:**
- ICE 스코어(Impact-Confidence-Ease): 3축 평가가 정밀하지만 Confidence 축이 실 데이터 없이 주관적이라 노이즈가 됨.
- MoSCoW(Must/Should/Could/Won't): 우선순위 분류는 가능하나 영향도와 난이도가 분리되지 않아 실행 순서 결정이 어렵다.

**근거:** 정적 사이트 개선은 대부분 HTML/CSS 수정이므로 Effort 판단이 직관적이고, 2축 매트릭스로 Quick Win을 즉시 식별할 수 있다.

### D3. 개선 실행 단위: 섹션 기반 + 크로스커팅 분리

**선택:** 개선 항목을 (1) 섹션 단위 개선과 (2) 크로스커팅 개선으로 분리한다.
- 섹션 단위: Hero 카피 정교화, Credibility 섹션 재배치, FAQ 보강 등 — 각각 독립 실행 가능.
- 크로스커팅: CTA 문구/디자인 통일, 폼 `name` 속성 추가, 모바일 전환 UX — 전체 페이지에 걸친 변경.

**근거:** 섹션별 카피/레이아웃 개선은 병렬 실행이 가능하지만, CTA 문구 통일이나 폼 속성 수정은 전체에 걸쳐야 하므로 별도 단위로 관리해야 충돌 방지와 일관성 확보가 된다.

### D4. 측정 전략: 이벤트 기반 + 스크롤 깊이 복합

**선택:** (1) CTA 클릭 이벤트 추적, (2) 폼 제출 시도 이벤트 추적, (3) 섹션별 스크롤 도달률 추적을 핵심 KPI로 정의한다.

**대안 검토:**
- 페이지뷰 + 이탈률만 추적: 단일 페이지이므로 이탈률이 무의미하고 세밀한 퍼널 분석 불가.
- 히트맵 도구(Hotjar 등) 도입: 유용하나 외부 서비스 의존이 Non-Goal 범위의 셋업을 요구.

**근거:** `js/main.js`에 이미 IntersectionObserver 패턴이 존재하므로 동일 방식으로 섹션별 스크롤 이벤트를 추가하면 최소 코드로 측정 가능하다. GA4 이벤트로 전송하면 별도 백엔드 없이 측정 완료.

## Risks / Trade-offs

- **[분석 주관성]** 퍼널 진단이 분석자 경험에 의존 → specs에서 체크리스트 기반 진단 항목을 명시하여 기준을 객관화한다.
- **[실제 데이터 부재]** 현재 트래픽/전환 데이터가 없어 Impact 판단이 추정에 기반 → B2B 랜딩 페이지 업계 벤치마크(평균 전환율 2-5%)와 휴리스틱 원칙으로 보완하고, 개선 후 데이터 축적을 통해 재조정한다.
- **[폼 미작동]** 현재 폼이 실제 전송되지 않으므로 전환율 측정 자체가 불가 → 폼 기능 정상화(name 속성 + 실제 엔드포인트)를 Quick Win 최우선 항목으로 배치하고, 측정 KPI는 폼 제출 버튼 클릭(시도)까지만 프론트엔드에서 먼저 추적한다.
- **[범위 확장 위험]** "마케터 관점"이 광범위하여 SEO, 콘텐츠 마케팅, 광고 최적화로 확장될 수 있음 → Non-Goals 경계를 엄격히 유지하고, 추가 영역은 별도 OpenSpec change로 분리한다.
- **[플레이스홀더 연락처]** 전화번호(`010-0000-0000`)와 카카오톡 링크가 더미 값 → 실제 연락처 업데이트는 운영자 확인이 필요하므로 tasks에서 "운영자 확인 필요" 태그를 부여한다.
